\chapter{Diseño}

Este capítulo describe las decisiones de diseño del sistema: cómo están organizados
sus módulos, cómo se estructura la base de datos, cómo se comunican los componentes
a través de la API REST, y cómo se ve la interfaz desde la perspectiva del investigador.
También incluye el diseño del pipeline de análisis conductual: el flujo de procesamiento
de video, la estrategia de separación por sujeto y la arquitectura del clasificador.

Los diagramas de este capítulo siguen la notación \textit{Unified Modeling Language}
(UML) versión~2.5.1, especificada por el \textit{Object Management Group}
(OMG) \cite{omg_uml}, organismo que gestiona y publica las actualizaciones del
estándar desde 1997. UML está también adoptado como estándar internacional bajo
ISO/IEC~19501:2005 \cite{iso19501}, que define la notación y el metamodelo para
diagramas de casos de uso, secuencias y clases. La estructura arquitectónica del
sistema se documenta conforme a ISO/IEC/IEEE~42010:2011 \cite{iso42010}, norma que
establece cómo describir una arquitectura de software mediante múltiples vistas
---estructural, de comportamiento y de despliegue--- que atienden las preocupaciones
de los distintos actores del sistema.

La decisión de diseño más importante fue separar el worker del backend en un proceso
independiente. La opción más simple habría sido ejecutar el pipeline dentro del mismo
servidor Flask, pero eso significa bloquear todas las peticiones HTTP mientras un video
de 20 minutos se procesa. El worker resuelve ese problema a costa de mayor complejidad
en el despliegue: hace falta una cola de tareas, un proceso adicional y comunicación
entre procesos. Esa complejidad extra está justificada porque el laboratorio puede
tener varios análisis en cola al mismo tiempo.

\section{Arquitectura general del sistema}
% -----------------------------------------------------------

El sistema tiene una arquitectura cliente-servidor de tres capas separadas en
contenedores Docker independientes: frontend, backend y base de datos. Un cuarto
contenedor ---el worker--- ejecuta el pipeline de análisis de forma asíncrona, sin
bloquear al servidor web mientras procesa los videos.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{figures/mermaid/arquitectura}
\caption{Arquitectura general del sistema [autoría propia].}
\label{fig:arquitectura}
\end{figure}

La Figura~\ref{fig:arquitectura} muestra la arquitectura. Las responsabilidades de
cada componente son:

\begin{itemize}
  \item \textbf{Frontend (React.js + Vite):} corre en el navegador del investigador.
        Se comunica con el backend exclusivamente mediante la API REST usando JSON.
        No tiene acceso directo a la base de datos ni al sistema de archivos del
        servidor.

  \item \textbf{Backend (Flask):} expone la API REST, gestiona la autenticación con
        JWT, valida los archivos subidos, encola las tareas de análisis y devuelve
        los resultados al frontend. Usa SQLAlchemy como ORM para interactuar con
        PostgreSQL.

  \item \textbf{Worker (Python):} proceso independiente que consume tareas de la
        cola, ejecuta el pipeline completo de análisis (preprocesamiento, detección,
        seguimiento y clasificación) y escribe los resultados en la base de datos.
        Al correr en un proceso separado, el servidor web sigue respondiendo a
        otras peticiones mientras el video se procesa.

  \item \textbf{Base de datos (PostgreSQL):} almacena usuarios, experimentos,
        sesiones de video, resultados por rata y conducta, y reportes generados.
        Los archivos de video se guardan en el sistema de archivos del servidor, no
        en la base de datos.
\end{itemize}

La separación en contenedores cumple con el atributo de portabilidad de la
ISO/IEC/IEEE~12207:2017 \cite{iso12207}: el sistema puede desplegarse en cualquier
equipo con Docker instalado sin dependencias adicionales. La separación entre worker
y backend también satisface el RF-12 (análisis automático en segundo plano) y permite
escalar el número de workers de forma independiente si el volumen de análisis aumenta.

\section{Diseño del pipeline de análisis conductual}
% -----------------------------------------------------------

El pipeline recibe un video del FST y produce como salida los tiempos totales de cada
conducta por espécimen y el desglose por minuto. Está organizado en cinco módulos
secuenciales: preprocesamiento, detección de ROIs, seguimiento, clasificación de
conducta y generación de resultados. La Figura~\ref{fig:pipeline} muestra el flujo
completo con las tecnologías de cada módulo.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{figures/mermaid/pipeline}
\caption{Diagrama de flujo del pipeline de análisis conductual [autoría propia].}
\label{fig:pipeline}
\end{figure}

\subsection{Módulo 1: Preprocesamiento}

El preprocesamiento usa \textbf{OpenCV VideoCapture} para decodificar el video
cuadro a cuadro. Si el contraste promedio cae por debajo del umbral mínimo, se
aplica \textbf{CLAHE} (\textit{Contrast Limited Adaptive Histogram Equalization})
\cite{bradski2000} para normalizarlo. Si tras la corrección el contraste sigue
siendo insuficiente, el pipeline se detiene y genera un reporte de diagnóstico
(ver RN-11).

\subsection{Módulo 2: Detección de ROIs}

El sistema detecta los cilindros en el encuadre usando contornos sobre los primeros
cuadros del video tras construir un modelo de fondo. Para cada cilindro calcula una
\textbf{región de interés} (ROI) que delimita el área de análisis de cada espécimen.
La \textbf{línea de agua} de cada cilindro se estima y estabiliza con un filtro de
\textbf{media móvil exponencial} (EMA) para reducir el efecto de reflejos y
variaciones de iluminación entre cuadros.

\subsection{Módulo 3: Seguimiento}

Dentro de cada ROI, el seguimiento combina detección y asociación temporal:

\begin{enumerate}
  \item \textbf{YOLOv8 + ByteTrack} \cite{zhang2022_bytetrack}: el detector
        YOLOv8 localiza al espécimen en cada cuadro usando el modelo
        \texttt{rat.pt}; ByteTrack asocia las detecciones entre cuadros
        consecutivos mediante IoU con filtro de Kalman, manteniendo la
        trayectoria durante oclusiones breves.
  \item \textbf{Asignación a ROIs por IoU máximo}: cada trayectoria activa
        se asigna a la ROI con la que tiene mayor solapamiento, separando
        espacialmente los especímenes aunque sus trayectorias se crucen.
  \item \textbf{Validación de consistencia}: comprueba que las posiciones
        predichas sean coherentes con el cuadro anterior (sin saltos abruptos
        o cambios anómalos), descartando detecciones inconsistentes.
  \item \textbf{Manejo de oclusiones}: cuando un espécimen desaparece
        temporalmente de la cámara, conserva su identidad y posición estimada
        hasta que vuelve a ser detectado dentro de su ROI.
\end{enumerate}

\subsection{Módulo 4: Clasificación de conducta}

El clasificador recibe el recorte de imagen alrededor del espécimen, normalizado
a $224 \times 224$ píxeles, y produce tres probabilidades: nado activo
(\textit{swim}), inmovilidad (\textit{immobile}) y escalamiento (\textit{escape}).
Opera en tres niveles según la confianza de la predicción (el umbral exacto
se definirá durante la fase de validación en TT-II):

\begin{enumerate}
  \item \textbf{ResNet-18} preentrenada en ImageNet con \textit{transfer learning}
        \cite{he2016,pan2010}: clasificador principal. Si la confianza de la
        predicción es suficiente, se acepta directamente.
  \item \textbf{ResNet-50} (fallback): si ResNet-18 no alcanza la confianza
        mínima, se consulta ResNet-50, una arquitectura más profunda con mayor
        capacidad de discriminación para casos ambiguos.
  \item \textbf{Heurística motion + aspect ratio}: si ambas redes quedan por
        debajo del umbral, se aplica una regla basada en el desplazamiento entre
        cuadros y la relación de aspecto del bounding box para distinguir
        inmovilidad de nado.
\end{enumerate}

Las predicciones individuales se consolidan con \textbf{voto mayoritario} en una
ventana deslizante de un segundo, reduciendo el efecto de clasificaciones ruidosas
en cuadros aislados. En caso de que el clasificador no alcance el F1 objetivo
diferenciando escalamiento de nado activo, el pipeline agrupa ambas conductas en una
categoría unificada denominada \textit{conducta activa} y lo registra en el reporte
(ver RN-13).

\subsection{Módulo 5: Resultados}

Al finalizar el análisis, el pipeline produce cuatro salidas:

\begin{itemize}
  \item \textbf{Tiempos totales} por espécimen: \texttt{swim\_s},
        \texttt{immobile\_s} y \texttt{escape\_s} en segundos, almacenados
        en la tabla \texttt{RESULTADOS\_COMPORTAMIENTO}.
  \item \textbf{Video anotado} (\texttt{\_tracked.mp4}): video original con
        las cajas de seguimiento y etiqueta de conducta superpuestas,
        disponible para revisión visual del investigador.
  \item \textbf{Desglose por minuto} (\texttt{per\_minute.json}): tiempo de
        cada conducta agrupado por minuto, almacenado en
        \texttt{COMPORTAMIENTO\_POR\_MINUTO}.
  \item \textbf{Datos de seguimiento} (\texttt{\_tracking.json}): trayectorias
        y confianzas cuadro a cuadro, útiles para depuración y auditoría del
        análisis.
\end{itemize}

\subsection{Separación por espécimen}

Los cilindros se graban en una sola toma cenital. La asignación de cada ROI a un
espécimen usa su posición espacial en el encuadre: se ordena primero por fila
(arriba/abajo) y luego por columna (izquierda/derecha), produciendo una numeración
consistente entre sesiones mientras el encuadre no cambie. Si el número de
cilindros detectados es menor al esperado, el sistema reporta cuáles especímenes
pudo procesar y cuáles no, en lugar de abortar el análisis completo.

\section{Diseño de la base de datos}
% -----------------------------------------------------------

El modelo de datos sigue el paradigma relacional. La Figura~\ref{fig:er} muestra el
diagrama entidad-relación físico con las tablas, columnas y claves del sistema.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{figures/mermaid/er}
\caption{Modelo entidad-relación físico del sistema [autoría propia].}
\label{fig:er}
\end{figure}

Las tablas están organizadas en cuatro grupos funcionales:

\textbf{Usuarios y acceso.}
\begin{itemize}
  \item \textbf{USUARIOS}: credenciales (correo, hash bcrypt), nombre, estado
        activo/inactivo y bandera \texttt{debe\_cambiar\_contrasena} que obliga
        al cambio en el primer ingreso cuando la cuenta fue creada por el
        administrador. El campo \texttt{ultimo\_acceso} permite auditoría de uso.
\end{itemize}

\textbf{Experimentos y material de video.}
\begin{itemize}
  \item \textbf{EXPERIMENTOS}: metadatos del experimento (nombre, fecha,
        tratamiento, especie, disposición y notas) con referencia al usuario
        propietario. Todos los investigadores pueden consultar cualquier
        experimento del laboratorio (ver RN-08).

  \item \textbf{VIDEOS}: hasta dos videos por experimento (campo \texttt{dia}:
        \texttt{dia1} o \texttt{dia2}), ruta de almacenamiento, duración en
        segundos y fecha de eliminación programada (30 días tras completar el
        análisis, ver RN-06).

  \item \textbf{SUJETOS}: un registro por espécimen dentro del experimento,
        identificado por su índice numérico y una etiqueta descriptiva.

  \item \textbf{ROIS}: región de interés (\textit{bounding box}: x, y, w, h)
        asignada a cada espécimen en cada video. La usa el pipeline para
        delimitar el área de seguimiento de cada animal.
\end{itemize}

\textbf{Análisis y trabajos.}
\begin{itemize}
  \item \textbf{CONFIGURACIONES\_ANALISIS}: parámetros del pipeline en el
        momento del análisis: nombre y hash del modelo, versión del pipeline,
        umbrales de confianza, inmovilidad, desplazamiento y aspecto de
        trepada. Permite reproducir o comparar resultados entre versiones del
        modelo.

  \item \textbf{TRABAJOS}: ejecución del pipeline sobre un video con una
        configuración específica. Registra estado (\texttt{en\_cola},
        \texttt{procesando}, \texttt{completado}, \texttt{error}), etapa
        actual, porcentaje de progreso, marcas de tiempo de inicio y fin, y
        mensaje de error si lo hay.

  \item \textbf{ANIMALES}: tabla de enlace entre un trabajo y un sujeto.
        Cada fila representa un espécimen procesado dentro de una ejecución
        concreta del pipeline.
\end{itemize}

\textbf{Resultados, reportes y notificaciones.}
\begin{itemize}
  \item \textbf{RESULTADOS\_COMPORTAMIENTO}: tiempo total en segundos de cada
        conducta (nado, inmovilidad, escalamiento) por animal dentro de un
        trabajo. Es la fuente principal para los reportes.

  \item \textbf{COMPORTAMIENTO\_POR\_MINUTO}: desglose temporal del mismo
        animal, con una fila por minuto analizado. Permite el gráfico de
        evolución temporal en la pantalla de resultados.

  \item \textbf{REPORTES}: referencia a los archivos PDF, CSV y XLSX
        generados, con su ruta y fecha de creación. Los reportes se conservan
        aunque el video original haya sido eliminado (ver RN-06).

  \item \textbf{NOTIFICACIONES}: avisos al investigador sobre eventos del
        sistema (análisis completado, error en el pipeline, alerta de disco)
        vinculados al usuario y, cuando aplica, al experimento correspondiente.
\end{itemize}

El diccionario de datos completo con tipos, restricciones y descripciones de cada
columna se incluye en los anexos.

\cite{omg_uml,iso19501}, usando actores, casos de uso y relaciones de

del sistema desde la perspectiva del usuario.

\textbf{Administrador}. El Investigador gestiona sus experimentos y consulta
resultados; el Administrador gestiona usuarios y monitorea el estado del sistema.
El Administrador hereda todas las capacidades del Investigador sobre los experimentos
de cualquier usuario (ver RN-08).

\subsection{Notificaciones}

\section{Diagrama de clases}
% -----------------------------------------------------------

El diagrama de clases \cite{omg_uml,iso42010} (Figura~\ref{fig:clases}) muestra
las entidades principales del dominio y sus relaciones desde la perspectiva del
backend, representando la vista estructural del sistema conforme a
ISO/IEC/IEEE~42010:2011. Las clases centrales
son \textbf{Experimento}, \textbf{Video} y \textbf{Trabajo}: un experimento agrupa
uno o dos videos; cada video origina un trabajo que ejecuta el pipeline. Las clases
\textbf{Sujeto} y \textbf{Animal} separan la identidad del espécimen (definida al
crear el experimento) de su participación en una ejecución concreta del análisis,
lo que permite reutilizar la misma definición de sujeto en distintas corridas. Los
resultados se desagregan en dos clases: \textbf{ResultadoComportamiento} para los
totales por espécimen y \textbf{ComportamientoPorMinuto} para el desglose temporal.
La clase \textbf{ConfiguracionAnalisis} registra los hiperparámetros del pipeline
en el momento de cada análisis, de modo que los resultados sean reproducibles aunque
el modelo o los umbrales cambien en versiones futuras.

laboratorio no tiene casos de uso donde cancele un análisis a mitad de camino, y

limpiar los archivos parciales. Si en el futuro surge esa necesidad, el endpoint
